\begin{abstract}
This report presents SEARCH 22.5 (Spectral Embedding Archive for Rational Crease Pattern Hyperspace), an inverted design methodology for the computational generation of uniaxial 22.5$^\circ$ origami crease patterns. Rather than solving a highly intractable nonlinear mixed-integer optimization problem for every novel user input, we shift the computational burden to an exhaustive precomputation phase. The pipeline enumerates valid topologies of axial creases using Boolean Satisfiability (SAT) solvers, constrains geometric degrees of freedom via Mixed-Integer Linear Programming (MILP) to optimize for crease pattern neatness, and constructs the crease pattern using a 4D rational coordinate basis to circumvent floating-point degradation. The resulting folded states then are mathematically sliced to extract their macroscopic uniaxial trees. By simulating discrete heat diffusion across these networks, the trees are embedded into 64-dimensional vectors using the Heat Kernel Trace (HKT) from spectral graph theory. At runtime, the inverse design problem is thus reduced to a FAISS (Facebook AI Similarity Search) nearest-neighbor query, allowing the system to instantly retrieve the precomputed crease pattern whose structural topology most closely matches the user's desired shape. This report serves as a comprehensive technical overview of the SEARCH 22.5 pipeline, detailing the algorithms, mathematical formulations, and implementation strategies employed at each stage of the process.
\end{abstract}


\section*{Introduction}

One of the biggest unsolved problems in origami design is the ability to systematically design models in the 22.5$^\circ$ angle system, in which creases meet at angles of multiples of 22.5$^\circ$. Many of the oldest known traditional origami designs use this angle system (including the fish, crane, and frog), as do some of the most popular modern-day designs, yet there remain no known mathematical methods to systematically design models within these angle constraints. Instead, most 22.5$^\circ$ origami designers rely on trial and error combined with an experienced artist's intuition.

In general, representational origami designs can be categorized into two categories: uniaxial and non-uniaxial. Uniaxial designs are those whose structure can be represented by a tree of flaps and rivers. Forward design with uniaxial methods, including circle packing, box pleating, and hex pleating, have been well-studied. In these systematic procedures, a tree is converted into a packing of circles, squares, or hexagons by formulating the problem an optimization problem defined by non-linear non-intersection constraints. Then, a crease pattern can be obtained from the packing by linear time deterministic procedures

It is theoretically possible to inject rigid integer constraints into the nonlinear solver to enforce the angle constraints of the 22.5$^\circ$ system. However, even if this massive mixed-integer nonlinear optimization problem is solved and a valid packing is found, there is no known way to convert the packing into a flat foldable crease pattern due to a phenomenon known as ``dense bouncing''. Unlike box pleating or hex pleating with rational quantized lengths, 22.5$^\circ$ angles generate lengths involving $\sqrt{2}$, for which no amount of subdivision will ever perfectly line up with a rational length, effectively requiring an infinite number of creases to resolve the packing. And unlike circle packing which solves this by placing creases at specific angles, 22.5$^\circ$ crease patterns are by definition restricted to creases at multiples of 22.5$^\circ$.

Conversely, generating a flat foldable crease pattern from a predefined tiling of 22.5$^\circ$ molecules is computationally straightforward through the use of straight skeletons. However, the inverse design of a tiling to acheive a desired tree has no known solution---in fact, prior to this project, there were no known implementations of the forward computation of extracting a tree from a tiling.


Therefore, we we exploit the computational tractibility of tilings. We first enumerate a massive database of valid 22.5$^\circ$ crease patterns based on axial tilings (Sections~\ref{sec:topology_enumeration} and~\ref{sec:tiling_realization}), then extract their underlying trees retroactively (Section~\ref{sec:cp_generation}). Finally, by combining the physics of heat transfer with a sophisticated database lookup algorithm (Section~\ref{sec:hkt}), we can approximate the user's inverse design problem of tree $\to$ crease pattern in near-constant time by simply searching for the closest match in the database.


\section{Database Generation Phase I: Topology Enumeration}\label{sec:topology_enumeration}
First, we must enumerate the valid arrangement of axial creases that define the tilings. A \textbf{topology} defines the angles and connective relationships of axial creases, disregarding exact metric lengths. Recall that ridge creases run at angle bisectors of the axial creases, and that all creases (including ridges) must be multiples of 22.5$^\circ$. Therefore, all axial creases must run at multiples of 45$^\circ$. We then exhaustively enumerate every valid topology on an $N \times N$ integer grid by modeling the problem as a Boolean Satisfiability (SAT) problem.

We embed a planar graph onto an $N \times N$ square grid. The boundary of the paper is fixed, comprising $4N$ predefined boundary segments. Inside the grid, every possible orthogonal and diagonal segment connecting adjacent integer coordinates is treated as a potential axial crease. We assign a boolean decision variable $e \in \{0, 1\}$ to every internal edge.

The total number of internal boolean variables for an $N \times N$ grid is precisely $4N^2 - 2N$. The SAT solver's task is to sort through the $2^{4N^2 - 2N}$ possible assignments to find unique topologies that satisfy the following rules. 

\subsection{Local Z3 Geometric Constraints}
To prune the inherently exponential search space, we restrict the Z3 solver using four core local constraints that physically validate the crease structure. These constraints are formulated in Conjunctive Normal Form (CNF)---a product of sums---using standard boolean algebra notation (where $\overline{x}$ is NOT, $+$ is OR, and multiplication is implicit AND).

\paragraph{Planarity Constraint}
An axial topology must be planar. The two diagonals within any single $1 \times 1$ cell would cross if both were present. Therefore, they are mutually exclusive. For the two diagonal boolean variables $d_1$ and $d_2$ in every cell $(x,y)$, we assert the single CNF clause:
\begin{equation}
    (\overline{d1_{x,y}} + \overline{d2_{x,y}})
\end{equation}

\paragraph{No Dangling Edges Constraint}
A crease cannot terminate abruptly in the middle of a sheet of paper. Define the \textit{degree} of a vertex as the number of active edges incident to it. Because the boundary perimeter is pre-populated as constantly active edges (value $= 1$), every vertex in the grid must have a total degree either $=0$ (completely inactive internal node) or $\geq 2$ (active continuous crease). For every node $v$ and its set of incident edges $E_v$, we apply a cardinality constraint:
\begin{equation}
    \sum_{e \in E_v} e \neq 1
\end{equation}

\paragraph{No T-Junction Constraint}
To prevent dense bouncing and other straight skeleton complications, we enforce a strict T-junction rule. For every internal node $(x,y) \in [1, N-1]^2$, we evaluate its 8 potential neighbors, grouped into four opposite pairs: $P_1 = (R, L)$, $P_2 = (U, D)$, $P_3 = (UR, DL)$, and $P_4 = (UL, DR)$. Let the two boolean variables representing the edges of pair $k$ be $a_k$ and $b_k$. If \textit{any} pair $i$ is continuously active ($a_i b_i$), then \textit{all} pairs $j$ intersecting at that node must share identical boolean states ($a_j = b_j$). To express this in CNF, for every pair $i \in \{1..4\}$ and every pair $j \in \{1..4\}$, we assert:
\begin{equation}
    (\overline{a_i} + \overline{b_i} + a_j + \overline{b_j}) \cdot (\overline{a_i} + \overline{b_i} + \overline{a_j} + b_j)
\end{equation}

\paragraph{Optional Symmetry Constraint}
We can optionally enforce global symmetry by identifying symmetric edges via a coordinate transformation $T$. For every internal edge $e$ and its transformed counterpart $e_{sym}$, we enforce equivalence ($e = e_{sym}$). In CNF, this expands to two clauses:
\begin{equation}
    (e + \overline{e_{sym}}) \cdot (\overline{e} + e_{sym})
\end{equation}

\subsection{Global Topological Constraints}
Once Z3 yields a satisfying assignment based on local constraints, we extract the resulting graph $G=(V,E)$ and pass it through a Python-level global topology filter before accepting it into the database.


\paragraph{Connectivity Constraint}

First, we require the graph to be a single connected component. Disconnected components represent polygons floating in space within other polygons, which causes severe problems during crease pattern generation. This is easily filtered using NetworkX's built-in connectivity check.

\paragraph{Unique Normals Constraint}

Second, we validate the internal faces. We utilize a Doubly-Connected Edge List (DCEL) half-edge traversal to extract all faces of the graph: for every vertex $v$, its incident neighbors are sorted angularly using a standard $\text{atan2}(dy, dx)$ function. Walking counter-clockwise around the leftmost turns allows us to systematically extract the closed loops that define the internal polygonal faces.

For each extracted face, we calculate the directional vector 
\[\vec{d} = (\Delta x / \gcd(\Delta x, \Delta y), \Delta y / \gcd(\Delta x, \Delta y))\]
of every boundary segment. We strictly enforce that a valid topological face must consist of sides with unique directions (ignoring sequential collinear segments). While this allows concave polygons, it limits the number of sides to 8 and ensures they do not zig-zag or ``gerrymander'', which would also make crease pattern generation significantly more difficult.

With these local boolean constraints and global topology filters, the SAT solver is able to enumerate all valid topologies.

\section{Database Generation Phase II: Tiling Realization}\label{sec:tiling_realization}

The output of the topology enumeration phase is an unweighted planar graph with vertices only temporarily placed on an integer grid. We now seek to convert each topology into a \textbf{tiling}, where the axial polygons are fully constrained and defined by 22.5$^\circ$ references. This process must optimize for human-foldability of the crease pattern. This is, after all, the original purpose of 22.5$^\circ$ crease patterns over ``easier'' design methods.

If a topology contains $n$ nodes, we theoretically possess $2n$ degrees of freedom in the Euclidean plane (moving each node in $x$ and $y$). Our objective is to apply constraints until all degrees of freedom are removed, and the tiling is fully locked into a single crease pattern with fully defined references.

\subsection{Basic Geometric Constraints}\label{sec:basic_constraints}
We begin by establishing the absolute requirements of the tiling. 

\subsubsection{Angle preservation constraints}
Every edge in the initial grid topology is aligned to a multiple of 45$^\circ$. As the vertices move to form the exact tiling, these edges must remain parallel to their original orientations. Let $\vec{u}$ and $\vec{v}$ be the position of two vertices connected by an edge, and $\hat{n} = (n_x, n_y)$ be the known normal vector of that edge. We enforce the following angular constraint for every edge:
\begin{equation}\label{eq:edge_angle}
    (\vec{v}-\vec{u}) \perp \hat{n}\implies {\hat{n} \cdot (\vec{v} - \vec{u})} = 0 \implies n_x v_x - n_x u_x + n_y v_y - n_y u_y = 0
\end{equation}
Note that this constraint is linear and homogeneous.

\subsubsection{Boundary conditions}
To prevent rigid-body translation and scaling, the bottom-left node is locked to the $(0,0)$, and the top-right node is locked to $(1,1)$. Note that the latter is an inhomogeneous constraint, which prevents the solver from collapsing all vertex coordinates to 0.

\subsubsection{Symmetry constraints}
If the topology was generated with a symmetry constraint, we apply the corresponding symmetry constraints to the coordinates. For example, for book symmetry across the vertical axis $x = N/2$, every pair of symmetric nodes $u$ and $u^*$ must satisfy:
\begin{equation}
    u_x + u^*_x = 1 \quad \text{and} \quad u_y - u^*_y = 0
\end{equation}
Note that this introduces another inhomogeneous constraint. For diagonal symmetry across $y = x$, every pair of symmetric nodes $u$ and $u^*$ must satisfy:
\begin{equation}
    u_x - u^*_y = 0 \quad \text{and} \quad u_y - u^*_x = 0
\end{equation}

We can then combine these linear constraints into a single matrix form $A\vec{x} = \vec{b}$, where $x = {[x_1, y_1, x_2, y_2, \ldots, x_n, y_n]}^T$ is a vector containing all node coordinates, each row of $A$ corresponds to one of the aforementioned linear constraints, and $\vec{b}$ is a vector of constants (mostly zeros, except for the few inhomogeneous conditions).

\subsection{Quadruplet equidistance constraints}
While only the basic geometric constraints are absolutely necessary, we can ``spend'' the remaining degrees of freedom to constrain the tiling into a crease pattern that is as ``nice'' as possible. It is difficult to quantify how ``nice'' a crease pattern is, but a good heuristic is to minimize the number of internal vertices generated by the straight skeletons of the polygons. This is because internal vertices of a straight skeleton are generally not flat foldable, so each internal vertex forces a propagation of creases that make the crease pattern increasingly messier. More internal vertices result in more hinge creases.

Internal vertices in a straight skeleton appear where 3 or more edges are equidistant from a common point. We can ``collapse'' internal vertices together by forcing $> 3$ sides to be equidistant from a common point. Therefore, we define a \textbf{quadruplet} as a set of 4 edges (not necessarily contiguous) on a polygon, which we can constraint to be equidistant from a shared incircle.

Let $\hat{n}_1, \hat{n}_2, \hat{n}_3, \hat{n}_4$ be the inward-facing unit normal vectors of the four edges in the quadruplet. Let $\vec{v}_1, \vec{v}_2, \vec{v}_3, \vec{v}_4$ be the coordinates of one of the two vertices on each edge. Let $\vec{p} = (p_x, p_y)$ and $r$ be the unknown position and radius of the shared incircle.

For the incircle to be tangent to all four edges simultaneously, the perpendicular distance from the center $\vec{p}$ to each edge $i$ must be exactly equal to $r$. Using vector projection, we can express the distance to a line as:
\begin{equation}
    \hat{n}_i \cdot (\vec{p} - \vec{v}_i) = r \quad \text{for } i \in \{1, 2, 3, 4\}
\end{equation}

We can rearrange this equation to separate the unknown variables ($\vec{p}$ and $r$) from the variables defined by the graph's geometry ($\vec{v}_i$):
\begin{equation}
    \hat{n}_i \cdot \vec{p} - r = \hat{n}_i \cdot \vec{v}_i
\end{equation}

This gives us a system of 4 linear equations. We can represent this system in matrix form as $Q \vec{\xi} = \vec{w}$, where $\vec{\xi}$ contains our unknowns:
\begin{equation}
    \underbrace{
    \begin{bmatrix}
    n_{1x} & n_{1y} & -1 \\
    n_{2x} & n_{2y} & -1 \\
    n_{3x} & n_{3y} & -1 \\
    n_{4x} & n_{4y} & -1
    \end{bmatrix}
    }_{Q}
    \underbrace{
    \begin{bmatrix}
    p_x \\
    p_y \\
    r
    \end{bmatrix}
    }_{\vec{\xi}}
    =
    \underbrace{
    \begin{bmatrix}
    \hat{n}_1 \cdot \vec{v}_1 \\
    \hat{n}_2 \cdot \vec{v}_2 \\
    \hat{n}_3 \cdot \vec{v}_3 \\
    \hat{n}_4 \cdot \vec{v}_4
    \end{bmatrix}
    }_{\vec{w}}
\end{equation}

Notice that $Q$ is a $4 \times 3$ matrix. We have 4 equations but only 3 unknowns ($p_x, p_y$, and $r$). For an overdetermined system to have a valid solution, the right-hand side vector $\vec{w}$ must lie perfectly within the column space of $Q$. We do not actually need to solve for $\vec{\xi}$. We only care that such an incircle exists. If we can eliminate $\vec{\xi}$ entirely, we are left with a relationship that the vertices $\vec{v}_i$ must satisfy.
Recall that a vector $\vec{w}$ is in the column space of $Q$ if and only if it is orthogonal to the left null space of $Q$. Let $\vec{w}^T = \begin{bmatrix} w_1 & w_2 & w_3 & w_4 \end{bmatrix}$ be a basis vector for the left null space of $Q$, such that:
\begin{equation}
    \vec{w}^T Q = \vec{0}^T
\end{equation}
Because the normals $\hat{n}_i$ are strictly fixed constants in the 22.5$^\circ$ system, the matrix $Q$ is constant. Therefore, $\vec{w}$ is also constant and can be precomputed.

For the incircle to exist, we must satisfy:
\begin{equation}
    \vec{w}^T \vec{\xi} = w_1 (\hat{n}_1 \cdot \vec{v}_1) + w_2 (\hat{n}_2 \cdot \vec{v}_2) + w_3 (\hat{n}_3 \cdot \vec{v}_3) + w_4 (\hat{n}_4 \cdot \vec{v}_4) = 0
\end{equation}
By expanding the inner dot products $\hat{n}_i \cdot \vec{v}_i = n_{ix}x_i + n_{iy}y_i$, we group the terms by the coordinate variables of our vertices:
\begin{equation}\label{eq:quadruplet_constraint}
    \sum_{i=1}^{4} \left( (w_i n_{ix}) x_i + (w_i n_{iy}) y_i \right) = 0
\end{equation}
Note that this is a strictly linear, homogeneous constraint acting entirely on the node positions $x_i$ and $y_i$, which can also be expressed as a row in the constraint matrix $A$ without ever solving for the position or radius of the incircle. For example, to add a quadruplet constraint to matrix expanded in Equation~(\ref{eq:Axb}), we would append the following row to $A$ and $b$ based on the normals of the rectangle:

In general, constraining a quadruplet (that contains at least one unconstrained edge) will remove one degree of freedom. However, due to internal symmetries across the tiling, it is possible for a single quadruplet constraint to collapse multiple straight skeleton vertices at once, while still only spending one degree of freedom. Therefore, we want to strategically select quadruplets that maximize the number of straight skeleton vertices they collapse. More importantly, it is possible for a quadruplet constraint to be satisfied but not actually reduce the number of vertices in the straight skeleton. This is because a vertex only appears in the straight skeleton if and only if it is strictly closer to its constituent edges than to any other edge in the polygon. If another edge ``invades'' the incircle, the quadruplet constraint is effectively useless. 

Even worse, it is possible for a quadruplet constraint to cause an edge to collapse to 0 or negative length, violating the planarity of the tiling. Or, the mouth of a concave polygon might clip through the other end, forming an invalid ``bowtie'' shape. These effects are defined by inequalities. Therefore, we must resort to a Mixed-Integer Linear Programming (MILP) formulation to strategically select the optimal set of quadruplets that maximize the number of collapsed straight skeleton vertices while ensuring the physical foldability of the tiling.

\subsection{Optimal Quadruplet Selection}
To evaluate the physical consequences of activating different quadruplets, the solver must be able to solve the linear system from Equation~(\ref{eq:Axb}) for any possible combination of quadruplets, and evaluate the validity of the resulting geometry. Let the state vector $\vec{X}$ be an augmented vector of the vertex coordinate vector $\vec{x}$:
\begin{equation}
    \vec{X} = \left[ \, \vec{x}, \; \vec{z}_{quads}, \; \vec{p}_{incircles}, \; \vec{r}_{incircles}, \; \vec{b}_{bowtie} \, \right]^T
\end{equation}
Where we concatenate the following vectors:
\begin{itemize}
    \item $\vec{x}$: A concatenation of continuous 2D coordinates $\vec{x}_i = (x_i, y_i)$ for every vertex in the graph
    \item $\vec{z}_{quads}$: Boolean decision variables $z_k \in \{0, 1\}$ denoting whether quadruplet candidate $k$ is enforced.
    \item $\vec{p}_{incircles}$: Continuous 2D coordinates $\vec{p}_k = (p_{k_x}, p_{k_y})$ representing the prospective incircle center for quadruplet candidate $k$.
    \item $\vec{r}_{incircles}$: Continuous variables $r_k \ge 0$ representing the prospective incircle radius for quadruplet candidate $k$.
    \item $\vec{b}_{bowtie}$: Boolean variables used to check topological validity of concave polygons.
\end{itemize}

Now recall that the standard form of a MILP problem is:
\begin{align}
    \max_{\vec{X}} \quad & \vec{c}^T \vec{X} \\
    \text{subject to} \quad & M \vec{X} \le \vec{B}
\end{align}
where $\vec{c}$ is the objective function as reward weights on the elements of $\vec{X}$, $M$ is the constraint matrix that encodes all of the linear equalities and inequalities, and $\vec{B}$ stores the constants on the right-hand side of each constraints.


Our objective is to maximize the number of quadruplets that can be activated without violating geometric constraints. Therefore, $\vec{c}$ contains non-zero weights only for the decision variables $\vec{z}_{quads}$. We heavily reward resolving concave (reflex) vertices because they create disproportionately messier crease patterns if not collapsed into a clean internal vertex. Accordingly, quadruplets involving reflex vertices are assigned a priority weight ($c_k = 10.0$), while standard convex quadruplets receive a base weight ($c_k = 1.0$).

\subsubsection{Formulating the Constraint Rows}

\paragraph{1. Basic geometric constraints}
First, the same basic geometric constraints described in Section~\ref{sec:basic_constraints} are applied to the vertex coordinates. These constraints only depend on the vertex coordinates $\vec{x}$. These are the only equality constraints in this problem formulation.

\paragraph{2. Edge Integrity:}
To prevent edges from collapsing to zero or negative length (inversion), we enforce a minimum feature size, $\epsilon>0$. Let $\hat{t}$ be the unit tangent vector of an edge. For any directed edge from vertex $\vec{u}$ to vertex $\vec{v}$, the projected distance must remain at least $\epsilon$:
\begin{equation}
    \hat{t} \cdot (\vec{v}-\vec{u}) \ge \epsilon \quad \implies \quad - \hat{t}^T \vec{v} + \hat{t}^T \vec{u} \le -\epsilon
\end{equation}
Note that $\epsilon$ is not for numerical stability, but another metric of crease patern ``niceness''. If $\epsilon$ is too small, the resulting crease pattern may contain tiny creases that are possible but unpleasant to fold. If $\epsilon$ is too large, the solver may not be able to find any valid solutions. We empirically set $\epsilon = \frac{1}{4N}$, where $N$ is the grid size of the original topology.

\paragraph{3. Quadruplet Activation (Big-M Method):}
If $z_k = 1$, quadruplet $k$ must be constrained such that the distance from the incircle center $\vec{P}_k$ to the boundary edge originating at vertex $\vec{u}$ with inward normal $\hat{n}$ must exactly equal the radius $r_k$. However, if $z_k = 0$, we need the constraint to be relaxed. To encode this as linear rows in $M$, we choose a large constant $C\approx 3N$ and apply the following inequalities to a vertex $\vec{u}$ on each edge in the quadruplet:
\begin{align}
    \hat{n}_u^T \vec{p}_k - \hat{n}_u^T \vec{u} - r_k + C z_k \le C \\
    -\hat{n}_u^T \vec{p}_k + \hat{n}_u^T \vec{u} + r_k + C z_k \le C
\end{align}

\paragraph{4. Incircle invasion:}
Recall that a vertex only appears in the straight skeleton if and only if it is strictly closer to its constituent edges than to any other edge in the polygon, so if another edge ``invades'' the incircle, the quadruplet constraint is effectively useless. To prevent this, we must ensure that any invader edge $i$ maintains a normal projection distance from the incircle center $\vec{P}_k$ that is at least $r_k$:

\begin{equation}
    -\vec{n}_i^T \vec{p}_k + \vec{n}_i^T \vec{x}_i - r_k + C z_k \le C
\end{equation}
The addition of this constraint is why the MILP formulation requires $\vec{p}_k, r_k$ as state variables, while Equation~(\ref{eq:quadruplet_constraint}) did not.

\begin{figure}[h]
    \centering
    \includegraphics[width=\textwidth]{figs/incircle_invasion.pdf}
    \caption{From left to right: (1) An unconstrained polygon and its messy straight skeleton. (2) Applying an incircle constraint to a quadruplet collapses two interior points into one. (3) If an edge from the same convex chain invades the incircle, the constraint is useless and the straight skeleton is broken. (4) An edge from a different convex chain (separated by a concave vertex) has a normal projection that invades the incircle, but without breaking the straight skeleton.}
\end{figure}

However, this equation treats every side of the polygon as an infinite line, while incircle invasion only occurs if the actual line segment invades the circle. Formulating this stronger constraint woould introduce a quadratic term (Euclidean distance from the incircle center). Instead, we realize that edges that do not share a convex chain with any edge in the quadruplet are incapable of actually invading the incircle, and as a conservative heuristic, only apply the infinite line incircle invasion constraint to all other edges.

\paragraph{5. Anti-Bowtie Clearance:}
In some concave polygons, it is possible for a concave ``mouth'' vertex to clip through an edge on the opposite side of the polygon, as illustrated in Figure~\ref{fig:bowtie}. To prevent this, we can enforce that every reflex vertex must reside in at least one of the three safe zones defined by the edge's normal and tangent vectors.

For every reflex vertex $\vec{x}_c$ and a non-adjacent edge $(\vec{u}, \vec{v})$ with normal $\hat{n}$ and tangent $\hat{t}$, we create three boolean variables based on the vertex's position relative to the edge: 
\begin{itemize}
    \item $b_1$ = 1 if $\hat{n} \cdot (\vec{x}_c - \vec{u}) < 0$ (the vertex is behind the edge)
    \item $b_2$ = 1 if $\hat{t} \cdot (\vec{x}_c - \vec{u}) > 0$ (the vertex is to the left of the edge)
    \item $b_3$ = 1 if $\hat{t} \cdot (\vec{x}_c - \vec{u}) < 0$ (the vertex is to the right of the edge)
\end{itemize} 

\begin{figure}[h]
    \centering
    \includegraphics[width=0.8\textwidth]{figs/bowtie.pdf}
    \caption{Left: an unconstrained polygon. Center: the polygon with a ``bowtie'' violation. Right: the three safe zones defined by $b_1,b_2,b_3$ and the danger zone (red) where the constraint is violated, defined by the edge's vectors $\hat{n},\hat{t}$ and minimum clearance size $\epsilon$.}\label{fig:bowtie}
\end{figure}


We can enforce that the reflex vertex must reside in at least one safe sector by applying the constraint:
\[b_1 + b_2 + b_3 \ge 1 \implies -b_1 - b_2 - b_3 \le -1\]
The MILP solver gets to choose which of $b_1, b_2, b_3$ is true. Similar to the quadruplet toggles $z_k$, we can use big-M constraints to link these boolean variables to toggle which which safe zone the vertex is constrained to. For example, the corresponding constraint for $b_1$ would be:
\begin{equation}
    \hat{n} \cdot (\vec{x}_c - \vec{u}) \ge \epsilon - C(1 - b_1) \quad \implies \quad -\hat{n}^T \vec{x}_c + \hat{n}^T \vec{u} + C b_1 \le C - \epsilon
\end{equation}
In total, four constraint inequalities are added to the system for every pair of reflex vertex and non-adjacent edge. 

\subsubsection{The Compound Matrix Inequality}
By concatenating these varied constraints, we construct the master MILP matrix equation. The inequality system $M \vec{X} \le \vec{B}$ takes the following block matrix structure:\footnote{Technically, the basic geometric constraints should be in a separate matrix because they represent equalities rather than inequalities, though for the sake of visual compactness we can handwave that each equality can be represented as a pair of inequalities. In practice, all the constraints are handed off to \texttt{PuLP} as a black box to deal with internally.}

\begin{equation}\label{eq:milp}
\begin{array}{l}
    \text{Basic constraints} \\
    \text{Edge Integrity} \\
    \text{Quadruplets} \\
    \\
    \text{Incircle Clearance} \\
    \\
    \text{Anti-bowtie}
    \\
\end{array}
\underbrace{
\left[
\begin{array}{ccccc}
    A & 0 & 0 & 0 & 0 \\
    \hdashline
    M_{edges} & 0 & 0 & 0 & 0 \\
    \hdashline
    -N_{quad} & C \cdot I_{quad} & N_{incircle} & -I_{quad} & 0 \\
    N_{quad} & C \cdot I_{quad} & -N_{incircle} & I_{quad} & 0 \\
    \hdashline
    N_{invader} & C \cdot I_{inv} & -N_{incircle} & -I_{inv} & 0 \\
    \hdashline
    0 & 0 & 0 & 0 & -S_{sector} \\
    N_{bowtie} & 0 & 0 & 0 & C \cdot I_{bowtie} 
\end{array}
\right]}_{M}
\underbrace{
\begin{bmatrix}
    \vec{x} \\
    \vec{z}_{quads} \\
    \vec{P}_{incircles} \\
    \vec{r}_{incircles} \\
    \vec{b}_{bowtie}
\end{bmatrix}}_{\vec{X}}
\le
\underbrace{
\begin{bmatrix}
    \mathbf{0} \\
    \hdashline
    -\boldsymbol{\epsilon} \\
    \hdashline
    \mathbf{C} \\
    \mathbf{C} \\
    \hdashline
    \mathbf{C} \\
    \hdashline
    -\mathbf{1} \\
    \mathbf{C} - \boldsymbol{\epsilon}
\end{bmatrix}}_{\vec{B}}
\end{equation}

Where:
\begin{itemize}
    \item $M_{edges}$ applies the tangent vector projections ($\hat{t}^T$) mapping adjacent vertices to their required edge clearance.
    \item $N_{quad}$, $N_{invader}$, and $N_{bowtie}$ apply the corresponding inward normal and tangent projections ($\hat{n}^T, \hat{t}^T$) mapping vertices to their respective distance inequalities.
    \item $N_{incircle}$ projects the prospective incircle centers ($\vec{P}_k$) against those same normal vectors.
    \item $S_{sector}$ is a block summation matrix that groups and adds the three boolean sector variables $(b_1 + b_2 + b_3)$ for each reflex-edge pair.
    \item $I$ represents identity matrices of the appropriate dimensions mapping the scalar variables ($\vec{r}_{incircles}$ and the boolean toggles).
    \item $\mathbf{C}$, $\boldsymbol{\epsilon}$, and $\mathbf{1}$ represent column vectors populated dynamically with the respective constants.
\end{itemize}
In total, $M$ and $\vec{B}$ are composed of pre-computable constants for each topology. The problem in this form can be passed into a MILP solver like PuLP to constrain the maximum number of quadruplets while maintaining the geometric validity of the tiling.

\subsubsection{Diversity Threshold for Multiple Solutions}
A single topology can often be tiled in several geometrically distinct ways. To populate the database with diverse layouts from the same root topology, we loop the MILP solver. Upon finding an optimal set of active quadruplets $\mathcal{A}$, we log the configuration and inject a strict integer cut back into the solver:
\begin{equation}
    \sum_{i \in \mathcal{A}} z_i \le |\mathcal{A}| - d_{threshold}
\end{equation}
where $d_{threshold}$ is a hyperparameter defining the minimum diversity difference between solutions. This bans the previous solution and forces the MILP to swap at least $d_{threshold}$ quadruplets, nudging it to discover alternative geometric variations---though it is not guaranteed that the resulting tilings will be unique, even if the sets of selected quadruplets are different.

\subsubsection{Quadruplet Harvesting}

The solution to the MILP comes with a set of vertex positions that is guaranteed to be physically realizable (satisfying the constraints applied to $M$). However, it cannot guarantee that the tiling is fully locked. This is because it is computationally prohibitive to feed every possible quadruplet into the MILP (consider the fact that an $n$-gon contains $\binom{n}{4}$ possible quadruplets), so we initialize it with a heuristic ``harvest'' of quadruplets that are either contiguous or are the nearest edges to a concave pair of edges. As a result, the MILP might not have had access to a key quadruplet needed to fully constrain the tiling.

We can quickly check the rank of $A$ with the current selection of quadruplets by performing Gaussian elimination over a finite field $\mathbb{Z}_p$ with a large prime~\cite{rankcheck}. If the matrix is rank-deficient, the $Z_p$ elimination identifies exactly which columns (variables) lack pivots. We map these free variables back to their physical nodes. The missing quadruplets will likely involve these unconstrained nodes. We then initialize a depth-first search scavenger that will explore the exhaustive pool of unused quadruplet candidates, specifically filtering for quadruplets that involve the identified sliding nodes. It tentatively applies each quadruplet constraint and keeps ones that increase the rank until full rank is reached. Only then are the constraints actually applied and solved for the exact tiling geometry. 


\subsection{Rational 4D basis}\label{sec:4D}

Let us take a detour. Before solving the linear system to compute the final tiling node positions, we need to address the representation of vertices in the 22.5 angle system. In conventional crease pattern softwares, points are represented as Cartesian coordinates stored as a pair of floats. However, 22.5 triangles immediately generate irrational lengths, which in the $Ax=b$ formulation are introduced through $\sqrt{2}$ terms in the normal vectors of 45$^\circ$ creases. We will encounter numerical precision issues if we attempt to represent vertices using this conventional data structure. 

We first observe that all Cartesian coordinates in the 22.5 angle system can be represented analytically in the form $(A_x + B_x\sqrt{2} , A_y + B_y\sqrt{2})$, where $A_x, B_x, A_y, B_y$ are rational numbers \cite{degenerative-coordinates-22.5}. This suggests that we can represent points in the 22.5 angle system as a tuple of four fractions, which can avoid numerical precision issues by storing the vertex as a list of 8 integers.\footnote{It can be stored in as few as 5 integers by using a common denominator, but the 8 integer form is easier for operations and provides information about reference-finding for the vertex.} 

It would be possible to directly convert the 2D Cartesian coordinate $(A_x + B_x\sqrt{2} , A_y + B_y\sqrt{2})$ to the 4D rational coordinate $(A_x, B_x, A_y, B_y)$. This is essentially a 4D vector space where the 4 basis vectors project to the 2D plane as $\hat{\imath}, \sqrt{2}\hat{\imath}, \hat{\jmath}, \sqrt{2}\hat{\jmath}$, respectively. An example of a few vertices represented in this orthogonal basis is shown in Figure \ref{fig:orthogonal-basis}.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\textwidth]{figs/orthogonal_coords.pdf}
    \caption{A few example vertices represented in the orthogonal 4D basis projection.}\label{fig:orthogonal-basis}
\end{figure}

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{figs/hypercube_coords.pdf}
    \caption{A few example vertices represented in the hypercube 4D basis projection. The vertex $(1,1,-1,1)$ can be understood intuitively as the sum of the basis vectors $\hat{x}, \hat{y},$ a negative $-\hat{z}$ and a $\hat{w}$.}\label{fig:hypercube-basis}
\end{figure}

An alternative projection would be to project the 4D basis vectors to be uniform in length, but spaced 45 degrees apart from one another. This is reminiscent of the projection commonly used to represent a 4D hypercube in 2D space, and was initially proposed for representing 22.5 crease patterns by Lang et. al. in \cite{hyper-cps}. It was shown by Lang et. al. that rational 4D coordinates project uniquely to the 2D plane, and 2D 22.5 vertices "lift" uniquely into 4D. An example of a few vertices represented in this hypercube basis is shown in Figure \ref{fig:hypercube-basis}. Note that this basis is not fundamentally different than the orthogonal basis: instead of having a basis vector for the $\sqrt{2}$ length in either direction, $\sqrt{2}\hat{\imath}$ is represented as $\hat{y} - \hat{w}$ and $\sqrt{2}\hat{\jmath}$ is represented as $\hat{y} + \hat{w}$. Either projection scheme would work for the rest of our 22.5 architecture. For the rest of this paper, we will choose the hypercube projection. 

\subsection{Exact Precision Linear Solve}
During the MILP solve, the tiling constraints were formulated using floating-point arithmetic for fast linear algebra. This is sufficient for finding approximate solutions in order to select quadruplets. However, the final tiling must be solved with rational arithmetic to guarantee exact precision downstream. We now transition the system $A\vec{x} = \vec{b}$ into the 4D hypercube basis discussed in Section~\ref{sec:4D}.

For example, recall the angular constraint from Equation~(\ref{eq:edge_angle}), requiring the edge between node $\vec{u}$ and node $\vec{v}$ to remain orthogonal to the edge's fixed normal vector $\hat{n} = (n_x, n_y)$:
\begin{equation}
    n_x(v_x - u_x) + n_y(v_y - u_y) = 0
\end{equation}
In our 4D basis, the physical 2D coordinates of the nodes are derived from their 4D rational coordinates: $\vec{u}^{(4D)} = [x_u, y_u, z_u, w_u]^T$ and $\vec{v}^{(4D)} = {[x_v, y_v, z_v, w_v]}^T$. Substituting the hypercube projection mapping for the edge vector components gives:
\begin{align}
    v_x - u_x &= (x_v - x_u) + \frac{1}{2}((y_v-y_u) - (w_v-w_u))\sqrt{2} \equiv A_x + B_x \sqrt{2} \\
    v_y - u_y &= (z_v - z_u) + \frac{1}{2}((y_v-y_u) + (w_v-w_u))\sqrt{2} \equiv A_y + B_y \sqrt{2}
\end{align}
The normal vector $\hat{n}$ is similarly defined by its rational and irrational components as $n_x = a_x + b_x \sqrt{2}$ and $n_y = a_y + b_y \sqrt{2}$. Substituting these components into the dot product yields a polynomial expansion:
\begin{equation}
    (a_x + b_x \sqrt{2})(A_x + B_x \sqrt{2}) + (a_y + b_y \sqrt{2})(A_y + B_y \sqrt{2}) = 0
\end{equation}
When the algebra is expanded and simplified, the variables naturally separate into two groups: those multiplied by 1, and those multiplied by $\sqrt{2}$. It takes the form:
\begin{equation}
    C_{rational} + C_{irrational} \sqrt{2} = 0
\end{equation}
Crucially, because $\sqrt{2}$ is an irrational number, it is impossible for a non-zero rational number to cancel out a non-zero irrational number. Therefore, for the dot product to be exactly zero, both coefficients must independently evaluate to zero. This transforms our single 2D constraint into two separate linear equations over the rationals:
\begin{align}
    C_{rational} &= A_x a_x + 2 B_x b_x + A_y a_y + 2 B_y b_y = 0 \\
    C_{irrational} &= A_x b_x + B_x a_x + A_y b_y + B_y a_y = 0
\end{align}
By applying this transformation to every row in the system, we can convert the entire problem into a rational linear system (with twice as many unknowns and twice as many rows), which can be solved using a custom Gauss-Jordan elimination solver. The resulting rational coordinates are guaranteed to prevent numerical instability or floating-point errors propagating into the crease pattern.


\section{Database Generation Phase III: Crease Patterns and Trees}\label{sec:cp_generation}

The exact tiling fully constrains the axial creases of the crease pattern. Next, the rest of the crease pattern (CP) must be constructed by filling in the polygons with straight skeletons and enforcing flat-foldability. Then, the crease pattern must be folded into a flat state, in order to then extract the tree structure of the folded state. Note that the CP only needs to satisfy Kawasaki's theorem (which is a geometric constraint); the topological constraints relating to mountain-valley assignment or layer self-intersection have no effect on the tree extraction results.

\subsection{Straight Skeleton Generation}
To route the interior creases of the faces defined by the tiling, we utilize a straight skeleton algorithm. Rather than implementing a custom straight skeleton algorithm (which is prone to complicated edge cases) that operates directly on the exact rational coordinates, we adopt a hybrid floating-point approach. This allows us to leverage existing optimized libraries while still ensuring the final output is perfectly rational. 

First, we project the exact 4D vertices of a face into 2D Cartesian floats. However, standard geometric algorithms---such as the sweep-line methods commonly used for straight skeletons---are notoriously fragile when handling IEEE-754 floating-point arithmetic. Because these algorithms rely on strict spatial sorting, an error of even a fraction of a bit during intersection calculations can cause the algorithm to make inconsistent topological decisions, leading to catastrophic program failures or ``singularities'' \cite{kettner2008classroom}. 

To bypass these well-documented robustness issues, we apply an aggressive quantization filter. By truncating the floating-point coordinates to a fixed precision (empirically determined to be 3 decimal places), we force near-degenerate points to snap to a rigid grid.

We feed this sanitized floating-point polygon into an off-the-shelf library to extract the topological connectivity of the interior creases. To reclaim the exact precision of these internal nodes, we iteratively look at their resolved neighbors (starting from the polygon boundary, whose exact 4D coordinates are known), extract the 22.5$^\circ$ routing angles dictated by the floating-point straight skeleton, and compute the intersection of creases using Equation~(\ref{eq:4D_intersection}).

\subsection{Kawasaki Propagation and Hinge Routing}
The resulting straight skeletons are generally not flat-foldable on their own. Kawasaki's Theorem mandates that for an interior vertex to fold flat, the alternating sum of the angles between its incident creases must equal exactly $180^\circ$. This implies that any odd-degree vertex (excluding those on the outer paper boundary) are not flat-foldable, and can be resolved by adding or removing a single crease.

To enforce flat-foldability, we use a greedy Kawasaki-fixing algorithm utilizing recursive raycasting. The algorithm filters for odd-degree vertices that violate Kawasaki. All hinge creases are at 45$^\circ$ angles---therefore, for a target vertex, it checks each of the 8 directions for a potential direction that would satisfy Kawasaki's theorem if a hinge crease were added. For each direction, it tentatively raycasts a hinge in that direction and counts how many creases are propagated before the hinge collides with a boundary or another vertex. To minimize the number of added creases, the algorithm then commits the hinge direction that satisfies Kawasaki's theorem while introducing the fewest number of new creases.

    \centering
    \includegraphics[width=0.8\textwidth]{figs/recursive_raycast.pdf}
    \caption{An example of a Kawasaki-preserving raycast operation recursively generating multiple creases to preserve flat foldability. This operation introduces 4 new creases.}\label{fig:recursive-raycasting}
\end{figure}


\subsection{The Stacked Folded State Architecture}
To calculate the 2D folded coordinates, we construct a spanning tree of the faces\footnote{The CP, which is a planar graph, explicitly tracks topological faces using a standard Doubly-Connected Edge List (DCEL) algorithm, sorting half-edges counter-clockwise and discarding the infinite outer face using signed-area calculations.} of the crease pattern via a Kinematics Breadth-First Search (BFS). Starting from an arbitrary root face, we span the adjacency graph, recursively reflecting the exact 4D vertices of child faces across their connecting hinge edges.

However, calculating coordinate geometry alone is insufficient; we must also explicitly track the connectivity of the overlapping paper layers. To achieve this, we define a specialized data structure, \texttt{Fold225}, which separates the abstract 2D topology from the physical paper layers (\textit{instances}):
\begin{itemize}
    \item \texttt{vertices}: A globally unique list of exact 4D coordinates present in the folded state.
    \item \texttt{edges}: A globally unique list of line segments, defined as pairs of vertex indices.
    \item \texttt{faces}: A globally unique list of topological polygons, defined by an ordered cycle of edge indices. The relative order of the edges represents the continuous boundary of the face.
    \item \texttt{instances}: The physical paper layers. Each unique topological face $f$ corresponds to a stack of one or more physical instances. An instance is represented as an array of connection pointers, where the $k$-th pointer corresponds to the $k$-th edge of face $f$. Each pointer stores the address $(f_{other}, i_{other})$ of the neighboring instance across that specific edge, or \texttt{None} if the edge is a paper boundary.
\end{itemize}

    \caption{A visualization of the folded state data structure for a waterbomb base. (a) The coordinate geometry defining unique edges and vertices. (b) The three unique faces defined by edges. (c) The instances of those faces and their connectivity.}\label{fig:fold-structure}
\end{figure}

\subsection{Orthogonal Slicing and Tree Extraction}
With the paper successfully folded into a layered stack of 2D polygons, we must extract its uniaxial stick figure (the macroscopic tree). We execute an orthogonal slicing algorithm---conceptually analogous to a medical CT scan---to isolate the connected appendages. The mental model for this process is projecting the folded model onto a horizontal axis and sweeping vertical slice lines across it to reveal its cross-sectional connectivity.

First, we determine the optimal projection axis $\vec{a}$ by evaluating the primary 22.5$^\circ$ grid directions and selecting the axis that yields the maximum projected bounding box width. We project all vertices of the folded state onto $\vec{a}$, filter for unique values, and sort them to identify the critical hinge points ($h$) along the axis of the model.

We mathematically sweep a 1D slice line $\mathcal{L}(h)$ perpendicular to $\vec{a}$, executing a slice at every critical hinge projection. When a slice intersects the folded state, it splits a single face into two faces joined at an unfolded crease. Because reindexing the connectivity map of the folded state's instances is a severe bottleneck, we utilize an optimized C++ routine (\texttt{split\_and\_rebuild\_cpp}) to handle the topological bifurcation. The C++ engine performs the following operations:
\begin{enumerate}
    \item \textbf{Edge Splitting:} Intersected edges are split at the intersection point, registering new sub-segments into a global edge map to prevent duplicates.
    \item \textbf{Face Bifurcation:} The vertex loop of each face is analyzed. If a face contains vertices strictly on both sides of the slice line, it is bifurcated into a left chain and a right chain. A new internal crease edge is generated to seal the two new topological faces.
    \item \textbf{Pointer Resolution:} The engine duplicates the old physical instances into the new bifurcated faces. It maps the newly created internal creases to point to each other, seamlessly sewing the severed halves together. For external connections, it checks the split geometry of the neighboring instance to ensure the pointer is routed to the correct geometric half of the neighbor.
\end{enumerate}

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{figs/tree_slicing.pdf}
    \caption{The tree extraction slicing process. (a) The crease pattern. (b) The folded state with axis and orthogonal slices through vertex projections. (c) The sliced crease pattern (i.e., the packing). (d) Disconnected components of the crease pattern as a result of slicing. (e) The final extracted tree based on the connectivity between hinge sets and the face components. Not shown: degree-2 vertices are removed from the tree.}\label{fig:tree-slicing}
\end{figure}

With the model completely sliced at every critical hinge location, we extract the tree by analyzing the connectivity of the face instances. We build an adjacency graph of all instances, except we explicitly ignore connections across hinge-parallel (that is, axis-perpendicular) edges. By filtering out these specific connections, the instances naturally decouple into discrete connected components, which physically represent flaps and rivers spanning between the hinge slices.

The topological tree is then constructed by analyzing how these components interact at the hinge locations:
\begin{enumerate}
    \item \textbf{Edges:} Every connected component spans between a minimum and maximum hinge value ($h_{start}$ and $h_{end}$). Each component forms an \textbf{Edge} in our macroscopic tree (not to be confused with the edges of the folded state or the crease pattern) with its metric weight derived from the axial length $h_{end} - h_{start}$. 
    \item \textbf{Nodes:} At each hinge location $h$, we must determine how the incident components converge to form structural nodes. Crucially, we do not simply merge all components that share the same projected hinge coordinate. Instead, we construct a local adjacency graph: two components are connected if and only if they share a physical paper connection across a hinge-parallel edge located \textit{exactly} at $h$. Each connected cluster within this local graph becomes a distinct \textbf{Node} in the tree. This rigorous clustering ensures that independent branches overlapping in the 2D projection remain topologically separate unless physically joined.
    \item \textbf{Simplification:} Finally, the raw tree is simplified by collapsing and merging intermediate degree-2 nodes. This distills the microscopic complexity of the sliced layers into the final abstract, undirected metric graph representing the structural appendages of the origami model.
\end{enumerate}

\section{Spectral Shape Analysis and the Heat Kernel Trace}\label{sec:hkt}

To mathematically quantify the shape of an origami uniaxial tree, we utilize a concept from spectral graph theory known as the Heat Kernel Trace (HKT). The physical intuition behind this approach is straightforward: if we treat the tree as a network of physical, thermally conductive rods, the geometric shape of the network dictates how heat diffuses through it. Similar trees will distribute and dissipate heat in a similar pattern. By simulating the flow of heat over time, we can extract an embedding vector uniquely describing the tree's shape. The following is a crash course derivation for readers unfamiliar with spectral graph theory.

\subsection{The Continuous Heat Equation}
We begin with the classical 1D heat equation, which describes how the temperature $u(x, t)$ at position $x$ changes over time $t$:
\begin{equation}
    \frac{\partial u}{\partial t} = \alpha \frac{\partial^2 u}{\partial x^2}
\end{equation}
where $\alpha$ is the thermal diffusivity of the material. To understand why the second derivative governs heat transfer, consider its discrete approximation using a step size $\Delta x$:
\begin{equation}\label{eq:second_derivative}
    \frac{\partial^2 u}{\partial x^2} \approx \frac{1}{\Delta x} \left( \frac{u(x+\Delta x) - u(x)}{\Delta x} + \frac{u(x-\Delta x) - u(x)}{\Delta x} \right)
\end{equation}
Notice that this equation simply sums the temperature differences between a point and its immediate neighbors, divided by the step size (the spatial gradients). This makes intuitive sense: if a spot $u(x)$ is warmer than both its left and right neighbors, the second derivative will be negative, meaning the cold spot will cool down. 

We can extend this logic to higher dimensions. Instead of the second derivative, we generalize using the Laplacian operator $\nabla^2$. For example, in a continuous 2D domain, the Laplacian is defined as:
\begin{equation}
    \nabla^2 u(x,y) = \frac{\partial^2 u}{\partial x^2} + \frac{\partial^2 u}{\partial y^2}
\end{equation}
Again, the continuous Laplacian simply measures how much the temperature at a specific point deviates from the average temperature of its immediate surroundings.

\begin{figure}
    \centering
    \includegraphics[width=0.8\textwidth]{figs/heat_equation.pdf}
    \caption{(a) The continuous 1D heat equation: a point $x$ will heat or cool depending on its temperature relative to its neighbors. (b) The heat equation for a discrete graph: similarly, the rate of change of temperature at each node is determined by the sum of differences in temperature with its neighbors.}\label{fig:heat-equation}
\end{figure}

\subsection{The Graph Laplacian}
We can transition this continuous physical model to our discrete origami tree graph. Let $u_i$ be the temperature at node $i$. Following the logic of the continuous case, the rate at which heat enters node $i$ is simply the sum of the heat fluxes from all adjacent nodes $j$. According to Fourier's law of conduction, the flux from $j$ to $i$ is proportional to the temperature gradient $\frac{u_j - u_i}{L_{ij}}$, where $L_{ij}$ is the physical edge length between the nodes. Thus, neglecting the material constant $\alpha$, the heat flow into node $i$ is:
\begin{equation}\label{eq:discrete_heat}
    \frac{du_i}{dt} = \sum_{j \sim i} \frac{u_j - u_i}{L_{ij}}
\end{equation}
where $j\sim i$ means ``for all nodes $j$ connected to node $i$''. The right-hand side of this equation resembles Equation~(\ref{eq:second_derivative}), the sum of temperature differences between a node and its neighbors, weighted by the inverse of the edge lengths. This makes intuitive sense: if node $i$ is colder than its neighbors, it will heat up; if it's hotter, it will cool down. However, recall that the derivative of a continuous function like in Equation~(\ref{eq:second_derivative}) increases in accuracy as $\Delta x$ approaches zero. Therefore, we force $L_{ij}$ to be small by subdividing large edges of our origami tree, ensuring that our discrete graph model faithfully approximates the continuous physics analogy.

A careful observer might ask: in the 1D continuous approximation (Equation 2), the $\Delta x$ terms multiply to form a $\Delta x^2$ in the denominator. Why do we only use $1/L_{ij}$ for our graph, rather than $1/L_{ij}^2$? The answer lies in the distinction between a continuous space and a discrete graph. The first division by length defines the thermal gradient (the driving force of the heat). In a continuous system, the second division by length accounts for the infinitesimally small \textit{mass} (or control volume) of the point being heated. In standard spectral graph theory, nodes are treated as abstract points with uniform mass, so the edge weight simply represents thermal conductance ($w_{ij} = 1/L_{ij}$). 

Because this standard graph formulation drops the spatial volume term, a standard graph Laplacian is only a true approximation of the continuous continuous Laplacian when all edge lengths $L_{ij}$ are uniform lengths. This further motivates our decision to pre-process trees with edge subdivision, ensuring that edge lengths are not only small, but uniformly small.

\subsection{Matrix Formulation}
In the continuous heat equation, the Laplacian $\nabla^2$ is a negative semi-definite operator (heat diffuses from hot to cold). By convention in spectral graph theory, we flip the sign of the operator so that its eigenvalues are non-negative. Thus, we define the Graph Laplacian $L$ such that the heat transfer equation becomes:
\begin{equation}\label{eq:matrix ode}
    \frac{d\vec{u}}{dt} = -L \vec{u}
\end{equation}
where $\vec{u}(t) = {[u_1(t), u_2(t), \dots, u_N(t)]}^T$ is the vector of temperatures at all $N$ nodes. The graph Laplacian $L$ has the same effect as the continuous Laplacian $\nabla^2$, but in matrix form (acts on a vector of node temperatures rather than a continuous temperature function).

To satisfy Equation~(\ref{eq:discrete_heat}) under this flipped sign convention, the $i$-th row of the matrix multiplication $L\vec{u}$ must equal $\sum_{j \sim i} \frac{1}{L_{ij}} (u_i - u_j)$. We can explicitly construct the matrix $L$ to achieve this:
\begin{enumerate}
    \item For the off-diagonal entries (representing the $-u_j$ terms), we set $L_{ij} = -\frac{1}{L_{ij}}$ if nodes $i$ and $j$ are connected, and $0$ otherwise. This is exactly the negative weighted Adjacency matrix, $-A$.
    \item For the diagonal entries (representing the $+u_i$ terms), node $i$ must be multiplied by the sum of the conductances of all its connected edges. We set $L_{ii} = \sum_{j \sim i} \frac{1}{L_{ij}}$. This is exactly the weighted Degree matrix, $D$.
\end{enumerate}
Therefore, the Graph Laplacian takes the elegant matrix form:
\begin{equation}
    L = D - A
\end{equation}

\subsection{The Heat Kernel and Matrix Exponential}
Equation~(\ref{eq:matrix ode}) is a system of linear, first-order ordinary differential equations. Because $L$ is a matrix of constants, the solution to this differential equation takes the form of a matrix exponential:
\begin{equation}
    \vec{u}(t) = e^{-tL} \vec{u}(0)
\end{equation}
We define this exponential matrix as the \textbf{Heat Kernel}, $H(t) = e^{-tL}$. 

Physically, $H(t)$ defines the complete thermal distribution of the graph at time $t$, for any initial condition. The entry $H_{ij}(t)$ represents the exact amount of heat that will have diffused from node $i$ to node $j$ if a single, unit-mass of heat was injected at node $i$ at $t=0$. The diagonal entries $H_{ii}(t)$ represent the amount of heat \textit{remaining} at the source node $i$. At $t=0$, none of the initial heat has had time to diffuse, so $H_{ii}(0) = 1$. As $t \to \infty$, the heat completely disperses through the network until it's evenly distributed across all nodes: $H_{ii}(\infty) = 1/N$.

\subsection{Eigendecomposition}
Because $L$ is real and symmetric, the Spectral Theorem guarantees that it can be diagonalized into an orthonormal basis:
\begin{equation}
    L = \Phi \Lambda \Phi^T
\end{equation}
where $\Phi$ is an matrix of eigenvectors $[\phi_1, \phi_2, \dots, \phi_N]$ (which is orthogonal, so $\Phi^T  = \Phi^{-1}$), and $\Lambda$ is a diagonal matrix containing the corresponding eigenvalues $\lambda_1, \lambda_2, \dots, \lambda_N$. Recall that an eigenvalue and an eigenvector satisfy the equation $L\phi_i = \lambda_i \phi_i$. 

By definition, the matrix exponential is a Taylor series: $e^{-tL} = I - tL + \frac{t^2}{2!}L^2 - \frac{t^3}{3!}L^3 + \dots$ 
When we raise $L$ to a power, the internal orthogonal matrices gracefully cancel out (e.g., $L^2 = (\Phi \Lambda \Phi^T)(\Phi \Lambda \Phi^T) = \Phi \Lambda (\Phi^T \Phi) \Lambda \Phi^T = \Phi \Lambda^2 \Phi^T$). This cancellation holds for every term in the Taylor series, allowing us to factor the eigenvectors completely out of the exponential:
\begin{equation}
    e^{-tL} = \Phi e^{-t\Lambda} \Phi^T = H(t)
\end{equation}

\subsection{The Heat Kernel Trace $Z(t)$}
Rather than dealing with the full $N \times N$ matrix $H(t)$, we can compress the graph's global thermal profile into a single scalar function of time by taking the trace of the Heat Kernel:
\begin{equation}
    Z(t) = \text{tr}(H(t)) = \sum_{i=1}^N H_{ii}(t)
\end{equation}
Recall that $H_{ii}(t)$ is the amount of heat remaining at node $i$ at time $t$, assuming node $i$ alone was injected with a unit of heat For each node, $H_{ii}(t)$ takes the form of an exponential decay $e^{-t\lambda_i}$. Thus, $Z(t)$ is the sum of these decaying exponentials. We can prove this using the cyclic invariance of the trace operator, or in other words, the fact that $\text{tr}(ABC) = \text{tr}(BCA)$:
\begin{align}
    Z(t) &= \text{tr}(\Phi e^{-t\Lambda} \Phi^T) = \text{tr}(\Phi^T \Phi e^{-t\Lambda}) = \text{tr}(I e^{-t\Lambda}) = \text{tr}(e^{-t\Lambda}) \\
    Z(t) &= \sum_{i=1}^N e^{-t\lambda_i}
\end{align}
This formulation can be computed quickly. By computing the eigenvalues of $L$ and applying the exponential decay for some values of $t$, we can extract a compact embedding vector that captures the global shape of the tree. In the implementation, we compute the first 32 eigenvalues and apply the decay at 64 values of $t$ from $t=0.00001$ to $t=10$ along a logarithmic scale, resulting in a 64-dimensional embedding vector for each tree.

\subsection{Accelerated Similarity Search via FAISS}
The database contains hundreds of thousands of exact rational tilings. Comparing a user's drawn tree against every database entry in real-time using raw graph matching or full eigenvalue extraction would be too slow. Instead, we employ Meta's FAISS (Facebook AI Similarity Search) library, utilizing a highly optimized C++ \texttt{IndexFlatL2} index.

During database generation, the exact tree for every tiling is extracted and stored in the database. At the end of the generation process, the FAISS cache is initialized by converting the trees into 64-dimensional HKT signatures. Because certain dimensions of the HKT vector exhibit much higher absolute variance than others, we apply Z-score normalization across the entire database. For each time step, we compute the database mean $\mu$ and standard deviation $\sigma$ across all signatures at that time, and normalize the dataset. This ensures the Euclidean (L2) distance metric weights all dimensions of the signature equally (see Figure \ref{fig:normalized_decay} compared to Figure \ref{fig:heat_decay}). The resulting matrix is loaded into the FAISS index and saved to disk alongside a mapping dictionary that links FAISS array indices back to the original SQLite primary keys.

At query time, the user's drawn tree is processed through the same HKT pipeline and normalization steps to produce a query signature. This signature is passed into the FAISS index, which performs a nearest neighbor search to retrieve the closest matching (L2 norm) tree signatures. The corresponding tiling entries are fetched from the database and returned to reconstruct the crease patterns, folded states, and trees for visualization in the frontend interface.

\section*{Conclusion}
The SEARCH 22.5 pipeline circumvents the historically intractable problem of forward 22.5$^\circ$ origami design through exhaustive precomputation and spectral indexing. The pipeline successfully bridges multiple discrete phases of computation. First, the exponentially vast $O(2^{4N^2})$ geometric search space is rigorously pruned using Z3 Boolean Satisfiability (SAT) to isolate valid planar topologies. Next, Mixed-Integer Linear Programming (MILP) is deployed to strategically assign quadruplet equidistance constraints via Big-M inequalities, eliminating degrees of freedom and locking the topology into a rigid metric tiling that optimizes the neatness and foldability of the crease pattern. To guarantee exact precision without quadratic field extensions or floating-point degradation, the linear system is solved over a 4D rational basis. Straight skeleton generation enhanced by discrete quantization and recursive Kawasaki raycasting is used to construct the flat-foldable crease pattern. The physical paper layers are then mathematically folded and subjected to an orthogonal sweep-line slicing algorithm, which distills the local layer connectivity into a macroscopic uniaxial tree. Finally, the Laplacian eigendecomposition of this structural tree is used to calculate the Heat Kernel Trace (HKT), projecting the crease pattern's topological shape into an $\mathbb{R}^{64}$ vector space. This allows the user's inverse design query to be resolved in sub-millisecond time via FAISS nearest-neighbor matching.

\subsection{Results}
Screenshots of the final interface are shown in Figures \ref{fig:interface1} and \ref{fig:interface2}. The user draws a tree on the left, which is processed through the HKT pipeline to produce a query signature. The FAISS index retrieves the closest matching trees from the database, and the corresponding tilings are used to reconstruct the crease patterns and folded states, which are visualized on the right. The retrieved designs are previewed as crease patterns but can be expanded to show the full pipeline process from topology to HKT signature match.

It is acceptable that the tree lengths of the retrieved designs do not perfectly match the query tree. The nature of 22.5 is that flap lengths are quantized in the form of $A + B\sqrt{2}$, which is unlike box pleating (where users can specify integer lengths) or circle packing (where users can specify any real length). If a user wants one flap roughly twice as long as another, they do not know whether it is better to set $\times 2$ or $\times (-5 + 5\sqrt{2})$, when in fact one may lead to a much more elegant solution than the other. Similarly, it is also acceptable that there are short stub lengths present or missing from the retrieved trees. As long as the paper length (or in HKT terms, the thermal mass) is in the right general distribution, the artist can then use techniques like point splitting or mush to turn a long flap into a shorter flap with details. This ``handwaviness'' is the reason the user is not allowed to provide numerical input for tree lengths, but only a visual sketch.

Empirically, from a database of around 300k tilings, the query returns close matches for trees representing simple quadrapeds, such as the giraffe shown in Figures \ref{fig:interface1} and \ref{fig:interface2}. More complex queries with more branches and deeper hierarchies (e.g., a dragon) will return more distant matches, but the returned designs can still serve as a useful starting point for artists to iterate towards their desired design.


\subsection{Limitations and Future Work}
While SEARCH 22.5 provides a robust framework for algorithmic design, the methodology currently exhibits several notable limitations. Foremost, the database remains relatively sparse. The current archive of approximately 300,000 tilings represents a minuscule fraction of the infinite 22.5$^\circ$ design space. Relying on brute-force enumeration and constraint solving poses severe scalability limits for larger grid dimensions ($N > 5$) necessary for more complex designs. Future research must pivot toward establishing an analytical mapping between desired tree metrics and optimal constraint selection to bypass the need for exhaustive generation entirely. 

Furthermore, the current pipeline is strictly confined to uniaxial bases. Non-uniaxial 22.5$^\circ$ design entails profoundly more complex kinematic dependencies and remains an open mathematical challenge; for the foreseeable future, this design process will solely rely on human intuition. It must also be emphasized that even within the uniaxial domain, SEARCH 22.5 functions as an assistive computational tool rather than an autonomous designer. The database serves to return the nearest structural approximation to a user's query, which invariably necessitates extensive artistic tweaking and iterative refinement to arrive at a final origami design. Additionally, the formulation of the input tree itself requires significant domain expertise and artistic vision.

\subsection{Final Remarks}
The computational design of origami is an inherently interdisciplinary endeavor. The success of SEARCH 22.5 relies on seamlessly integrating concepts from a wide array of seemingly unrelated mathematical and computational fields. The pipeline relies on Boolean Satisfiability from formal logic; Mixed-Integer Linear Programming from operations research; Gaussian elimination over finite fields and 4D rational bases from abstract algebra; and straight skeleton generation from computational geometry. The ultimate indexing of these structures relies on Spectral Graph Theory and the physics of discrete heat diffusion to quantify abstract shapes, utilizing FAISS---the same vector-search technology powering YouTube's video recommendation system. By strapping together results from these diverse disciplines, SEARCH 22.5 demonstrates that the historically intuitive art of 22.5$^\circ$ design can be computationally conquered towards a more complete understanding of origami.

\subsection{Crease Intersections in 4D}
To intersect creases formed by lines originating from $\vec{v}_a$ and $\vec{v}_b$ at angles $\alpha$ and $\beta$, we don't solve using a system of equations---instead, it is faster to solve this problem geometrically. 
\begin{align}\label{eq:4D_intersection}
    \Delta i & = (v_{b_x} - v_{a_x}) + (v_{b_y} - v_{a_y} - v_{b_w} + v_{a_w})\frac{\sqrt{2}}{2} \\
    \Delta j & = (v_{b_z} - v_{a_z}) + (v_{b_y} - v_{a_y} + v_{b_w} - v_{a_w})\frac{\sqrt{2}}{2} \\
    \beta_i & = \frac{\Delta j - \tan(\alpha)\Delta i}{\tan(\alpha) - \tan(\beta)} \\
    \beta_j & = \tan(\beta)\beta_i \\
    v_c & = v_b + \beta_i \begin{bmatrix} 1 \\ 0 \\ 0 \\ 0 \end{bmatrix} + \beta_j \begin{bmatrix} 0 \\ 0 \\ 1 \\ 0 \end{bmatrix} \\
\end{align}

\subsubsection{Recursive ray casting and unfolding}
Ray casting begins with a vertex and an angle, and extends until the first crease or vertex it intersects. The nearest intersection can be found quickly without checking all edges because the nearest intersection must be along an edge that shares a face with the starting vertex. A visualization of this fact can be seen in figure \ref{fig:raycasting}. Although computing the face connectivity is $O(n)$ and checking intersections with all edges is also $O(n)$, precomputing the face connectivity only needs to be done once to make all ray casting efficient for that crease pattern. If the ray intersects an existing crease, we can preserve Kawasaki flat foldability by recursively reflecting the ray from the intersection vertex along the reflected angle, as demonstrated in figure \ref{fig:recursive-raycasting}. If the ray intersects a boundary edge or a higher degree vertex, the ray casting ends.

\section{Appendix: Database Architecture and Infrastructure}
To manage the generation of millions of crease patterns and their high-dimensional representations, SEARCH 22.5 implements a highly parallelized, decoupled database architecture. The generation pipeline is separated into two independent phases—Topologies and Tilings—designed to minimize inter-process communication and bypass disk I/O bottlenecks.

\subsection{Phase I: Search Space Pruning and Topology Generation}
As the origami grid area expands, the combinatorial search space $O(2^{4N^2})$ rapidly exceeds the capacity of a single SAT solver instance. To solve this, we implemented a massive multiprocessing architecture utilizing \textit{prefix seeding}.

We divide the global search tree by hardcoding the first $k$ edges to a specific binary prefix (e.g., a 16-bit sequence like \texttt{10110...}), leaving the solver to find valid configurations for the remaining edges. However, naively generating all $2^k$ prefixes results in millions of ``dead'' paths that inherently violate planarity or degree constraints within their first few bits. 

To solve this, we execute a \textit{Pre-Pruning Phase}. A miniature Z3 instance, constrained exclusively to the first $k$ variables, iteratively extracts valid models. Previously found prefixes are blocked by adding the logical negation of their exact bit-state back to the solver:
\begin{equation}
    \text{Block Clause} = \bigvee_{i=1}^k (v_i \neq b_i)
\end{equation}
This dynamically shrinks the search tree. For example, a 16-bit prefix sequence natively contains 65,536 combinations, but Z3 pre-prunes this down to a much smaller subset of viable, mathematically possible roots.

Parallel worker processes fetch these viable prefixes, initialize a fresh Z3 solver with the prefix hardcoded as absolute truth, and exhaustively discover every valid topology down that specific branch. Because the SAT solver inherently generates isomorphic duplicates (rotations or reflections), we canonicalize the geometries before saving. We apply all 8 rigid transformations of the $D_4$ group, strictly sort the edges, and select the lexicographically smallest edge tuple as the absolute representation. 

To minimize storage overhead, this canonical edge list is compressed into a contiguous binary byte-string and hashed into a 64-bit MurmurHash3 (\texttt{mmh3}) integer. Workers flush these states to SQLite utilizing a native \texttt{INSERT ... ON CONFLICT DO NOTHING} statement bound to the hash index. By shifting the deduplication burden down to the SQLite C-engine, the Python workers can asynchronously write thousands of states per second without expensive cross-process locks.

\subsection{Phase II: Tiling Generation and Smart Auditing}
The Tiling database is populated by an orchestrator that pulls pending topologies from Phase I. Because the MILP quadruplet solver can generate multiple geometrically distinct tilings from a single topology via diversity integer cuts, this database implements a one-to-many schema.

Each tiling entry requires roughly 2--4 kB of storage and contains:
\begin{enumerate}
    \item \texttt{hashed\_tiling}: A 64-bit hash of the canonicalized exact 4D coordinate geometry, preventing identical straight skeletons generated from different MILP branches from duplicating.
    \item \texttt{tiling\_blob}: A Pickled dictionary containing the exact $\mathbb{Q}^4$ vertices and topological faces.
    \item \texttt{embedding}: A serialized \texttt{NetworkX} graph of the resulting macroscopic tree.
\end{enumerate}

\textbf{Data Integrity and Smart Migration:} Over the course of the project, schema updates (such as shifting from raw \texttt{float32} numpy embeddings to full \texttt{NetworkX} tree representations) introduced legacy data fragmentation. Rather than executing days-long full database recomputes, the pipeline employs a ``lazy-loader'' smart auditing script. Utilizing Python duck-typing and targeted \texttt{pickle.loads()} \texttt{try/except} blocks, the pipeline rapidly sweeps the database in the main thread, instantly skipping healthy tree objects and pushing only corrupted or legacy byte-strings to the multiprocessing pool for full geometric reconstruction. 

\subsection{Accelerated Similarity Search via FAISS}
Comparing a user's drawn tree against hundreds of thousands of database entries in real-time requires bypassing the SQLite database entirely during the initial search. Unpickling complex geometric blobs from disk introduces unacceptable latency. 

To achieve instant results, we utilize Meta's FAISS (Facebook AI Similarity Search) library.
\begin{enumerate}
    \item \textbf{Offline Precomputation:} A caching script extracts the eigenvalues from the stored trees, computes their Heat Kernel Trace (HKT) signatures, and normalizes them via Z-scores (to ensure all dimensions of the trace are weighted equally regardless of variance). These signatures are loaded into a FAISS \texttt{IndexFlatL2} index and saved to disk alongside a lightweight mapping dictionary linking the FAISS array indices back to the original SQLite primary keys.
    
    \item \textbf{Federated Querying:} When a user submits a query, the input tree is immediately converted to an HKT signature, normalized using the cached $\mu$ and $\sigma$ parameters, and simultaneously searched across multiple active database configurations (e.g., $N=4$ none, $N=5$ diag).
    
    \item \textbf{The Uniqueness Filter:} A critical edge case in origami design is that geometrically distinct raw tilings can occasionally fold into visually identical Crease Patterns (CPs). If the pipeline simply requested the top $n$ nearest neighbors, the user might receive identical results. Instead, the pipeline deliberately requests a massive buffer from FAISS (10$\times$ the target result count). It sorts these candidate hits globally by L2 distance, dynamically reconstructing their CPs and canonicalizing them. If a CP has already been seen in the current query batch, it is discarded. The algorithm halts exactly when $n$ geometrically unique CPs have been found.
\end{enumerate}

\subsection{Client-Server Decoupling and Serialization}
To ensure the heavy geometric processing does not block the user interface, the frontend drawing application and the backend mathematical engine are strictly decoupled. 

The frontend is a lightweight JavaScript canvas where users draw standard graphs. The backend server translates this drawing into an abstract mathematical tree. Once the backend identifies the best matches via FAISS, it must transmit the results back to the browser. However, web browsers cannot process the exact rational algebra ($\mathbb{Q}^4$) objects or \texttt{NetworkX} graphs used by the geometric engine. 

The backend employs a dedicated serialization pipeline to bridge this gap. Crease pattern segments and folded state layers are systematically stripped of their high-precision rational data types, resolved into standard 2D floating-point arrays, and bundled into a pure-data JSON payload. This allows the frontend to rapidly iterate over basic numeric arrays to draw standard SVG polygons and lines, completely abstracting the complex topological mathematics away from the UI.
